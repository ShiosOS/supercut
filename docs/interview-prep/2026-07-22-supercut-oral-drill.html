A. Architecture under interruption

Speak for the full time. If you finish early, keep going into edge cases until the timer ends.

A1. In 90 seconds: what does Supercut do, for whom, and what is the one architectural rule?

Target: product + rule, no file names yet.

Model answer

Supercut turns a market name into a self-contained HTML playbook of what’s working in that market’s long-running paid social video ads — counted patterns, hook frames, three tests. Built for a creative strategist briefing a shoot, not a data dump.

Rule: AI describes single ads; code finds patterns; AI explains patterns. Video model never sees two ads; prose model never sees a video — only tallies.

  • Follow-up: “So is AI doing the analysis?” → No. AI perceives and phrases; analysis is TypeScript tallies with receipts.
  • Follow-up: “What’s a receipt?” → Every count keeps contributing ad ids; exemplars link to live permalinks + frames.

A2. Walk the pipeline stages in order. For each stage say: inputs, outputs, AI or code, one failure mode.

3 minutes. Don’t peek at architecture.md.

Model answer

            fail: API success:false body, credits, empty angles
triage      → reordered Ad[]          | AI captions only
            fail: chunk error → keep unsure order (never reject)

A3. Draw (or narrate) what is on disk after a successful coffee scan. What’s never on disk?

2 minutes.

Model answer

Under data/coffee/: searches/*.json (raw list responses, params-hash keyed), factSheets/<adId>.json (schemaVersioned), mechanicals.json (fingerprint + cuts + threshold), frames/<adId>/n.webp, details/<adId>.json for exemplars, tallies.json, progress.json. Playbook at examples/coffee.html.

Never persisted: the mp4s — temp dir inside withVideo, deleted in finally.

  • Follow-up: What invalidates a fact sheet? → _schemaVersion !== FACT_SHEET_SCHEMA_VERSION or Zod parse fail.
  • Follow-up: What invalidates mechanicals? → stored sceneCutThreshold ≠ current constant.
  • Follow-up: Why round lastSeenAfter to UTC day? → stable cache key; second-precision would miss cache and re-spend credits.

B. “Why this number?” constant defense

Interviewers love constants. For each, say the value and the recon/reason — not just “it felt right.”

B1. Defend SCENE_CUT_THRESHOLD = 0.2. What happened at 0.4?

Model answer

UGC jump cuts (same room, camera shifts) score ~0.2–0.35, below a “true” scene change near 1.0. At 0.4, ffmpeg reported zero cuts on 40 of 96 real ads that visibly had them. 0.2 matched hand-checked counts. Chapters print ffmpeg cuts, not the model’s estimate; QA reports MAE vs model.

B2. Why WATCH_CONCURRENCY = 32? What becomes unsafe if you naively parallelize?

Model answer

Watch is network-bound (model call dominates). Sequential was 15–25 minutes; parallel cold scan <1 minute. Unsafe: concurrent RMW on mechanicals.json. Fix: load Map once, mutate in memory, persist once. Also: one ffmpeg pass per ad for all hook frames so CPU doesn’t melt at 32-wide.

B3. Why engagement uses a rate with a 10k play floor and shares×5 — and why longevity still weighs 0.5?

Model answer

Plays are bought → raw counts rank budgets. Rate = (likes + shares×5)/plays asks what viewers did once reached. Share is stronger endorsement than like. Floor kills 180/200 fake-high rates; Facebook often has 0 plays → engagement component 0, rank on longevity + format confidence. Longevity leads (0.5) because it’s the admission thesis; engagement breaks ties (0.3); format confidence (0.2) keeps shaky labels out of exemplar strips. Formula printed in playbook footnote.

B4. Why six search angles and exactly 240 searched? Why include ad-spend sort (21)?

Model answer

One sort is one bias. Four strata on broad keyword (engagement, delivery days, last found, ad spend) + two sub-keyword pulls × 40 = 240 = credit ceiling. Ad spend strengthens admission from “ran a long time” to “ran a long time with real money behind it.” Like-rate sort was considered and cut — engagement already enters via study score with floor/weights; fifth pull is marginal diversity for 40 credits.

C. Failure & honesty scenarios

Answer as if debugging on a call. Name the file you’d open first.

C1. Coffee scan finishes with only 8 admitted ads and throws. What failed, where, what do you tell the user?

Model answer

scan.ts after gate: pool.length < MIN_ADS_FOR_PLAYBOOK (10) throws — refuse to write astrology. Progress stage failed with message suggesting broader market. First coffee recon had this shape before triage (14 relevant / 56 watched). Mitigations already in system: product pre-filter, triage ordering, sub-keywords. Open progress/error, rejection appendix reasons, triage logs.

C2. Re-run same day spends PipiSpy credits again. List three distinct bugs that could cause that.

Model answer

  1. Recency cutoff not rounded to UTC day → params hash changes every second.
  2. API key accidentally included in hashed params (code hashes body.params only — if someone “fixed” it to hash whole body, key/env changes bust cache).
  3. Constants changed (put_day_min, data_type, sort, page size, region) → new hash, intentional re-pull.
  4. Bonus: market slug mismatch writing to different data/<m>/searches tree.
  5. Bonus: detail calls outside free window for many exemplars (1 credit each) — different from list cache, but feels like “re-spend.”

C3. Appendix shows high cut MAE and many contradicted hooks. What do you trust in the playbook, and what do you discount?

Model answer

Trust: chapter cut medians (ffmpeg), admission metadata (days/plays from library), tallied enums that don’t depend on verbatim speech, rejection table (marketFit).

Discount: spoken hook quotes (and any prose leaning on them), model cut estimates (already unused in chapters), maybe format labels if low-confidence share is also high — those ads stay in counts but shouldn’t front exemplars (already filtered).

Point to appendix as designed honesty surface — this is a feature, not an embarrassment.

C4. Two ads look identical but both survive dedupe. How do you debug fingerprinting?

Model answer

Open mechanicals.json for both ids. Compare sha256 (byte-identical?), duration delta (>1.5s means never merge), perceptual hash Hamming distance (must be ≤5). If re-encoded with different length/crop/end-card, average-hash of middle frame may diverge — known limitation. Dedupe sorts by longevity and keeps longest-running when match. Tests in test/dedupe.test.ts encode the contract.

E. Hostile / tradeoff round

Answer the challenge first. Then reveal. Your answer should include what you give up.

E1. “Your longevity filter just selects ads from rich brands with patience. Isn’t this budget bias with extra steps?”

Model answer

Acknowledge: survival can include brand inertia / forgotten campaigns — that’s why one ad is never a winner. Mitigations: brand cap (4), multi-brand exemplar picks, spend stratum so “still delivering” ≈ money still paying, pool-wide counts not exemplar worship, median days live made visible per chapter. Alternative (plays/engagement sorts) is worse budget bias. Honest limit stated in README: survival ≠ profitable.

E2. “Why shouldn’t I build the whitespace finder? That’s what strategists actually want.”

Model answer

“Nobody does X” needs census-level coverage; sample of survivors can’t prove absence. Gaps mean tried-and-died as often as opportunity. We surface rarity as observation with counts (“watch out”), not advice to charge in. Credible future: sameness score via image-similarity search on exemplar hooks — positive evidence of crowding, not fake absence.

E3. “This app is toy ops — no queue, no retries. Defend that for a production mindset.”

Model answer

Scope: prove the document, not a job platform. One scan per market; progress file is enough UX. Retries exist where they matter (model validation once; per-ad watch failures become skips). What I’d add if productizing: job row + cancel, structured logging, credit budget governor, dead-letter for describe failures — without changing describe→tally→explain.

E4. “Zod on every model call is overhead. Why not trust response_format?”

Model answer

Providers still wrap fences / drift fields. One schema = TS type + JSON schema + runtime parse. Retry-with-error once is cheap vs poisoning tallies. Downstream pure functions assume validated FactSheets — that’s the invariant that makes tally/QA boring (good).

F. Whiteboard prompts

On paper or a blank note. 5–8 minutes each. Then compare.

F1. Sketch the sequence diagram for one ad from watchAd entry to cache write, including temp files.

Must include

  • durationSeconds early skip before download
  • cache hit path: factSheet + mechanicals threshold match → return
  • curl → temp mp4 → probe → size → fingerprint → cuts → frames → describe → Map.set → delete temp
  • error → plainSkipReason / detail split

F2. Write pseudocode for dedupeCreatives + capPerBrand. Then mutate: what breaks if you cap before dedupe?

Model answer

Cap-before-dedupe: a brand’s 4 slots can be filled by 4 near-duplicate reuploads; after dedupe you’d have 1 creative but you already dropped other brands’ ads… actually you cap the list you have — if you cap first, duplicates consume slots, then dedupe collapses them and the brand under-represents while other brands were cut. Dedupe-first maximizes distinct creatives, then cap limits brand dominance.

F3. Design the next feature from the README roadmap: “living playbook” with monthly diff. What stays invariant? What new state do you need?

Model answer

Invariant: describe→tally→explain; receipts; no winners.

New state: advertiser monitor subscriptions (~$1/advertiser/mo), snapshot tallies per month, diff of distributions (which counts moved), stable ad ids across runs, UI for “what changed.” Risk: regenerating prose every month without anchoring to prior chapter ids → noisy diffs; maybe diff tallies first, only re-explain when counts move past a threshold.
